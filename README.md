# BookReview

## 概要

BookReview は、書籍の登録・レビュー投稿・閲覧ができる Web アプリケーションです。
個人の読書記録だけでなく、複数ユーザーが本の感想を共有できるサービスを想定しています。

## 制作背景

私は Java / Spring Boot を用いた Web アプリケーション開発の理解を深めるために、本アプリを作成しました。

単に CRUD 処理を実装するだけでなく、Controller / Service / Repository の責務分離、Spring Security による認証、JPA によるデータ永続化など、実務に近い構成を意識しました。

また、レビュー投稿という題材を選ぶことで、書籍とレビューの関連、ユーザー認証、管理者機能など、Web アプリケーションに必要な基本要素を段階的に学べると考えました。

開発環境では H2 Database を使用し、本番環境では PostgreSQL / Neon に接続する構成を想定しています。

## 主な機能

- 書籍一覧表示
- 書籍詳細表示
- 書籍登録・編集・削除
- レビュー投稿
- ユーザー登録
- ログイン・ログアウト
- 管理者権限による管理機能

## 工夫した点

### Spring Security による認証機能

ユーザー登録、ログイン、ログアウト機能を実装しました。
また、一般ユーザーと管理者ユーザーを区別することで、将来的に管理者のみが利用できる機能を追加できる構成にしました。

### 開発環境と本番環境の DB 切り替え

開発時は H2 Database を使用し、本番環境では PostgreSQL / Neon を使用する構成を想定しています。
環境変数を用いてデータベース接続情報を管理することで、機密情報をコードに直接書かないようにしました。

### Controller / Service / Repository の責務分離

アプリケーションの保守性を高めるため、Controller に処理を集中させず、Service 層を設けました。
Controller は HTTP リクエストの受け取りと画面遷移、Service は書籍やレビューに関する処理、Repository はデータベースアクセスを担当するように分けています。

## 苦労した点・解決したこと

### Spring Security の理解と実装

本アプリの開発で特に苦労したのは、Spring Security を用いた認証機能の実装です。
開発前にも書籍で Spring Security について学習していましたが、当時は設定項目や認証処理の流れを十分に理解できず、深く学ぶ前に一度飛ばしてしまっていました。
しかし、BookReview にログイン機能を追加する過程で、ユーザー登録、ログイン、ログアウト、権限管理などを実際のアプリケーションに組み込む必要があり、改めて Spring Security と向き合うことになりました。
最初は書籍や Qiita などの記事を参考にコードを書いても、自分のアプリの構成に合わせるとうまく動かないことが多くありました。また、AI にエラー内容やコードを投げて修正案を得ても、そのままでは解決しないこともありました。
そのため、単にコードを写すのではなく、SecurityConfig、UserDetailsService、PasswordEncoder、ログイン後の遷移先などがそれぞれ何を担当しているのかを確認しながら修正を繰り返しました。
現時点で Spring Security を完全に自力で実装できるとは言えませんが、開発を通して「認証処理がどのような流れで行われているのか」「どの設定がログイン処理や権限管理に関係しているのか」を以前より具体的に理解できるようになりました。

### 実装したい機能を設計に落とし込む難しさ

開発を進める中で、作りたい機能のイメージをそのままコードにすることの難しさを感じました。
例えば、「レビューを投稿できるようにしたい」「ログインしたユーザーだけが特定の操作をできるようにしたい」といったアイデアは思いついても、それを Controller、Service、Repository、Entity、画面テンプレートのどこにどのように反映すればよいのかを考える必要がありました。
最初から一人で設計することは難しかったため、実装したい機能を AI に相談し、提案された方針を自分のアプリの構成に合わせて書き直しながら実装しました。
その過程で、機能を実装するには単にコードを書くのではなく、データの持ち方、画面遷移、入力フォーム、DB との関係、認証・認可の条件などを分解して考える必要があると学びました。
